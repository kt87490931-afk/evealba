## 2. 핵심 DB 및 플랫폼 컨셉

### 2.0 이브알바 핵심 설계 원칙

**핵심 DB 4가지** — 업종/직종, 지역/세부지역, 편의사항/키워드, **MBTI**

이 4개 조합이 이브알바의 **핵심 데이터 축**이며, 채용정보·인재정보 모두 이 구조를 공유합니다. MBTI는 **AI 근접 매칭**에 유용합니다.

**플랫폼 역할**: 구인구직의 **핵심 플래너** — 3개 조합으로 기업회원과 일반회원에게 정보를 **어떻게 잘 보여줄지**가 관건입니다.

### 2.0-1 필터링 전략 및 체크리스트 방향

| 구분 | 설명 | 구현 |
|------|------|------|
| **정확한 필터링** | 사용자가 선택한 조건과 **완전 일치**하는 결과만 노출 | 1차 구현 필수 |
| **근접한 필터링** | 유사·인접 조건까지 포함 (인접 지역, 유사 직종, 겹치는 키워드, **MBTI 그룹** 등) | 2차 고도화 |

**체크리스트 작성 필수 (AI 매칭 전제)**  
양쪽 체크리스트가 모두 완성되어야 AI 근접 매칭이 가능합니다.

| 회원 유형 | 작성 시점 | 체크리스트 요구사항 |
|-----------|-----------|----------------------|
| **기업회원** | **채용정보** 작성 시 | 체크리스트 **전체 필수** 작성 |
| **일반회원** | **인재정보** 작성 시 | 체크리스트 **전체 필수** 작성 |

- **일반회원**이 채용정보 **조회** 시: 내 인재정보 체크리스트(지역·직종·편의·키워드) 기준으로 채용공고 정확 매칭 → 근접 매칭 순 노출
- **기업회원**이 인재정보 **조회** 시: 채용정보 체크리스트(업소 조건) 기준으로 인재 정확 매칭 → 근접 매칭 순 노출

### 2.0-2 DB 설계 방향

- 업종/직종, 지역/세부지역, 편의사항/키워드, MBTI를 **마스터 테이블**로 분리
- 채용공고·인재 프로필은 마스터 ID 참조 (단일 진실 원천)
- 관리자 화면에서 항목 추가·수정 시 필터·매칭에 자동 반영

## 3. 구현가능성 분석

### 3.1 전체 평가: **가능 (중상급 난이도)**

원문의 "개발난이도 - 쉬움"과는 달리, 실제 구현 시에는 **중상급** 수준의 난이도가 예상됩니다.

| 기능 | 구현가능성 | 난이도 | 비고 |
|------|-----------|--------|------|
| 회원가입(Pass 인증) | ✅ 가능 | 중 | Pass API 연동 필요 |
| 기업/일반 회원 분리 | ✅ 가능 | 하 | 권한/그룹 분리 |
| 2차 인증 체크리스트 | ✅ 가능 | 중 | 폼 설계·DB 설계 |
| 지역별 실시간 채팅 | ✅ 가능 | 중 | ScorePoint `plugin/chat` 사용, 지역 분리·성별 제한 커스텀 |
| AI 매칭 시스템 | ⚠️ 조건부 | 상 | 알고리즘 설계 또는 LLM API 활용 |
| 자동 썸네일 생성 | ✅ 가능 | 중 | 이미지 처리·캐싱 |
| 자동 글 작성 | ⚠️ 조건부 | 중~상 | 템플릿 기반 vs AI 생성 |
| 배너·광고·점프 | ✅ 가능 | 중 | 결제 연동·스케줄러 |
| 열람권 구매 | ✅ 가능 | 중 | 결제·권한 체크 |

### 3.2 기능별 상세 분석

#### 3.2.1 Pass 인증 (미성년자 가입 차단)

- **방식**: 한국신용평가정보(KCB) 또는 NICE 등 본인인증 API
- **구현**: 가입 시 본인인증 필수, 생년월일로 미성년자 차단
- **예상 기간**: 1~2주 (API 신청·계약·테스트 포함)

#### 3.2.2 지역별 실시간 채팅 (여성만 이용)

- **채팅 위젯**: ScorePoint `plugin/chat` **그대로 사용**
- **레이아웃**: `left-login.php` + `plugin/chat` (chat_box.php) — ScorePoint와 동일 구조
- **추가 개발**: 지역 분리·성별 제한(여성만)은 `plugin/chat` 커스텀으로 확장

#### 3.2.3 AI 매칭 시스템

- **전제 조건**: 기업회원은 채용정보 작성 시, 일반회원은 인재정보 작성 시 각각 체크리스트 **전체 필수** 작성 → 체크리스트 매칭으로 AI 근접 매칭 가능 (§2.0-1)
- **일반회원**: 인재정보 체크리스트 기준으로 하루 3개 업소(채용공고) 추천
- **기업회원**: 채용정보 체크리스트 기반 인재 매칭 (1명/일, 유료 시 5명/일)
- **구현 방향**:
  1. **규칙 기반**: 키워드·지역·편의사항·**MBTI 그룹** 점수 매칭 → 구현 용이, 근접 매칭에 MBTI 활용 (§11.7-1)
  2. **ML 기반**: 유사도·추천 모델 → 데이터 축적 후 고도화
  3. **LLM API**: 요구사항 텍스트로 매칭 설명 생성 → UX 개선용 보조
- **예상**: 1차는 규칙 기반, 이후 데이터 기반으로 개선하는 단계적 접근 권장

#### 3.2.4 자동 썸네일 생성

- **방식**: 업로드 이미지 리사이즈·크롭 또는 OG 이미지 자동 추출
- **구현**: PHP GD/Imagick 또는 별도 이미지 서비스
- **예상 기간**: 1주 내외

#### 3.2.5 자동 글 작성

- **해석에 따른 구현**:
  - **A**: 템플릿 + 입력값 병합 → 쉬움
  - **B**: AI가 채용문/이력 스타일로 문장 생성 → LLM API 필요
- **권장**: A로 1차 구현, 필요 시 B 추가

#### 3.2.6 배너·점프·결제

- **점프 기능**: 해당 게시물을 최상단으로 이동, 쿨타임 10분, 자동 점프 옵션
- **결제**: 토스페이먼츠/이니시스/나이스페이 등 PG 연동
- **스케줄러**: Cron으로 기간 만료·자동 점프 처리

#### 3.2.7 회원 권한·글 작성 규칙

| 회원 유형 | 작성 가능 | 비고 |
|-----------|-----------|------|
| **일반회원** | 커뮤니티글만 | 채용공고 작성 불가 |
| **기업회원** | 채용공고 + (커뮤니티 등) | 결제 확인 전: 비공개 / 확인 후: 공개 전환 |

**채용공고 특이사항**

- 결제 확인 전: 채용공고 **비공개** (작성은 가능)
- 결제 확인 후: 채용공고 **공개** 전환
- 채용공고 **본인만 수정** 가능 (작성자 본인만)

#### 3.2.8 배너 시스템 (수동 등록 vs 랜덤 자동 생성)

- **수동 배너**: 유료회원이 직접 업로드 (PNG, GIF만 허용)
- **랜덤 생성 배너**: 무료 옵션, 선택 옵션 조합으로 자동 생성
- **노출 우선순위**: 수동 배너 > 랜덤 배너 (수동 없을 시 fallback)
- **유료 기간 만료 시**: 수동·랜덤 배너 모두 삭제 (파일 + DB)

## 10. 작성자 의견

### 10.1 난이도에 대한 의견

문서에는 "개발 난이도 - 쉬움"으로 적혀 있으나, 실제로는 **중상급**에 가깝습니다.

- Pass 본인인증·PG 결제·실시간 채팅은 각각 API 연동과 안정성 작업이 필요하고  
- AI 매칭은 1차 규칙 기반이라도 설계·검증이 필요하며  
- 점프·배너·열람권 등 수익 로직은 상태 관리가 복잡합니다.

### 10.2 구현 전략 제안

1. **MVP(최소 기능) 우선**
   - Pass 인증, 기업/일반 회원, 채용/인재 게시판, 기본 필터
   - 채팅: ScorePoint `plugin/chat` 그대로 사용 (left_login + 채팅위젯)

2. **AI는 2단계로**
   - 1차: 키워드·지역·편의사항 기반 단순 매칭
   - 2차: 사용 로그·피드백 기반 추천 고도화

3. **법률·운영 검토**
   - 성인 관련 서비스이므로 이용약관, 개인정보 처리, 연령 제한 정책을 반드시 검토하는 것이 좋습니다.

### 10.3 개인회원 매칭제공·제공정보 관련 의견

1. **24시간 제한** — 기업회원이 열람/쪽지 가능한 24시간이 지나면 자동으로 차단. DB에 `matched_at` 저장, Cron 또는 API 호출 시점 검증 권장.

2. **항목별 공개/비공개** — 제공정보 각 항목에 `show_to_corp` 플래그 두고, 기업회원 열람 시 공개된 항목만 노출. 기본값은 비공개로 두고 사용자가 개별 켜는 방식이 안전함.

3. **매칭 완성 전 무료** — 초기 단계에서 매칭 품질 검증·피드백 수집에 유리. 유료 전환 시점은 "매칭 품질 지표"와 "유저 만족도" 기준을 먼저 정의한 뒤 진행하는 것을 권장함.

4. **지역 3개** — 개인회원이 여러 지역 희망 시 유연하게 대응 가능. 다만 매칭 로직에서 "3개 중 1개라도 겹치면 매칭" vs "모두 겹쳐야 매칭" 규칙을 미리 정해두는 것이 좋음.

### 10.4 결론

- **구현가능성**: ✅ **충분히 가능**
- **권장 접근**: Phase 1~2로 핵심 기능을 먼저 완성하고, Phase 3에서 AI·광고·점프를 추가하는 순차 개발
- **우선순위**: Pass 인증 → 회원·게시판·제공정보 → 채팅 → 결제·광고 → AI 매칭 순으로 진행하는 것을 권장합니다.

- 폰트 아웃라인 크기 & 컬러

**구현 권장**: PHP Imagick, 출력 포맷 JPG 또는 WebP (처리 속도 우선)

### 13.2 수동 vs 랜덤 배너 노출 로직

| 조건 | 노출 |
|------|------|
| 유료 결제 후 + 수동 배너 없음 | 랜덤 배너 **1회 생성** 후 저장·노출 |
| 수동 배너 등록함 | 랜덤 배너 숨김, **수동 배너** 노출 |
| 수동 배너 제거함 | 저장해 둔 **랜덤 배너** 다시 노출 |
| 유료 기간 만료 | 수동·랜덤 배너 **모두 삭제** (파일 + DB) |

### 13.3 파일 포맷

| 구분 | 허용 포맷 | 비고 |
|------|-----------|------|
| **수동 등록** | PNG, GIF | 투명·애니메이션 지원 |
| **랜덤 생성** | JPG 또는 WebP | PNG 불필요, 빠른 처리 우선 |

### 13.4 동일 실수 방지 (배너)

| 상황 | 해결책 |
|------|--------|
| 랜덤 배너 매번 재생성 | 1회 생성 후 경로 저장, 수동 제거 시 저장본 재사용 |
| 유료 만료 후 파일 잔존 | Cron에서 만료 회원 조회 → 배너 파일 삭제 + DB NULL |
| 수동 업로드 포맷 오류 | 서버에서 PNG/GIF만 허용, 확장자·MIME 검증 |
| 72×33 한글 가독성 저하 | 폰트 크기·굵기 사전 테스트, 최소 글자 크기 제한 |
